Understanding File-level Architecture


In order to see how all of the objects, functions, and components fit together, it is necessary to examine the purpose of each of the files in the DB2 Health Monitor Sample Application. Files are broken down into four groups:

  1. Web Service Consumer GUI
  2. Web Service Consumer
  3. Web Service Producer
  4. Web Service Producer DB

Web Service Consumer GUI

All files grouped into this category are used to present the XML data retrieved by the Web service consumer into a meaningful format for the user. This includes HTML documents, CSS documents, JavaScript widgets, JavaScript XML parsing, and XMLHTTP (asynchronous) requests.

index.html

This page serves as the entry point to DB2 Health Monitor Sample Application for PHP. It is designed to present the user with the necessary fields to enter and submit login information to the Web Service Consumer.

main.html

This page serves as the AJAX-based central health monitoring console. The user will only see health monitoring data if they have logged in correctly, and all components of the Web service are functioning correctly. This is the page to which all JavaScript widgets and necessary style information are attached and it is the page from which XMLHTTP requests are made.

blank.html

This page is a blank HTML page with a black background that is used by the Dialog widget as a backdrop to make the content appear as if it is in the foreground of the browser window.

index_style.css

This file stores all style information for the index.html page.

main_style.css

This file stores all style information for the main.html page that is not widget related.

table_style.css

This file stores style information used to customize the look of all SortableTable widgets on the main.html page.

content_style.css

This file stores style information used to customize the look of all ContentBar widgets on the main.html page.

dash_style.css

This file stores style information used to customize the look of all Dashboard widgets on the main.html page.

title_style.css

This file stores style information used to customize the look of all TitleBar widgets on the main.html page.

header.js

This script declares and initializes all global variables used by the Web Service Consumer GUI JavaScript-based components.

parser.js

This script includes functions to initialize widgets, destroy widgets, parse the Web service XML data, and to build and display that data using the customized widgets.

updater.js

This script contains the function responsible for using the AJAXUpdate widget to request the next XML data update from the Web Service Consumer. It uses the URL of the Web Service Consumer to re-direct the request and append the definition and logout flags to the request URL, as a way to distinguish what type of data should be received or what action to perform.

Web Service Consumer

All files grouped into this category are related to the actual consumption of Web service XML data. This includes gathering required inputs for the Web service (for example, login information), validating the inputs, retrieving the Web service XML data, validating the data, and producing error messages where applicable.

PHP XSD XML
  • db2soa.soap.client.header.php
  • db2soa.soap.client.php
  • class.client.validator.php
  • class.client.error.php
  • DB2Health_v0_8.xsd
  • Config_v0_1.xsd
  • config.xml

db2soa.soap.client.header.php

This PHP script defines global variables used by the Web Service Consumer in order to communicate with the both the Web Service Producer and the Web Service Consumer GUI. These global variables include the names of HTML controls used on the index.html page from which Web service input information is extracted, the GUI pages to which to redirect to, and the URL of the Web service WSDL document as required by Web service communication.

db2soa.soap.client.php

This is a procedural PHP script that serves as the central document to which all Web Service Consumer scripts are attached and included. Additionally, it processes all Web service requests sent by the Web Service Consumer GUI, related to login, updates, and logout. The script utilizes the appended URL query parameters and the session ID in the client-side cookie to determine the user's login status, which Web service function was requested, and if health definitions are required. It is ultimately responsible for requesting and receving data from the Web service.

class.client.validator.php

This PHP class is responsible for validating all data received from the Web Service Consumer GUI, as well as all data received from the Web Service Producer. Login information, URL parameters, and XML data are all scanned for potential risks that could compromise the security of the Web service architecture whether they are received from the GUI, the consumer, or even the producer itself.

class.client.error.php

This PHP class serves as a self-contained error object with which all Web Service Consumer errors can be fully described. It defines all of the error codes and messages that the Web Service Consumer is capable of returning. To ensure server consistency, the error objects contain an XML string representation as defined in the Web service WSDL. By following this example, the Web Service Consumer GUI only has to recognize a single type of XML error format. It will also log all XML error messages to the server disk.

DB2Health_v0_8.xsd

This XML schema defines the XML format (structure) with which the Web Service Producer returns XML data. While this may not be provided by the company who hosts the Web Service Producer, the information necessary to create it can be extracted from the WSDL document. This is used by the Web Service Consumer as a validation mechanism to ensure that the Web Service Producer is returning data in the specified format.

Config_v0_1.xsd

This XML schema defines the XML format (structure) for the config.xml file used by the Web Service Consumer. The schema acts as a validation mechanism to ensure that the individual maintaining the config.xml file has entered the parameters correctly.

config.xml

This file stores information that couples the Web Service Consumer GUI login information to the Web Service Consumer as well as the address of the WSDL document for the Web service from which the data is being gathered. See Setting Up The Sample for further information.

Web Service Producer

All files grouped into this category are related to the creation of the Web service. This includes the WSDL document (that is, a function registry and XML schema information) and the shells for each of the exposed Web service methods.

PHP XML
  • db2soa.soap.server.header.php
  • db2soa.soap.server.php
  • db2soa.soap.server.functions.php
  • DB2Health_Report.wsdl

db2soa.soap.server.header.php

This PHP script defines global variables that are used by the Web Service Producer including the location of the DB2Health_Report.wsdl document as well as the XML return type used by each of the functions, as defined in the WSDL.

db2soa.soap.server.php

This PHP script serves as the central document to which all Web Service Producer scripts are attached and in which the Web service SOAP server is created. It includes all necessary code to use the WSDL document to obtain the required health information from the database. It is the place where all of the functions are registered to be exposed, as defined in the WSDL document.

db2soa.nusoap.server.functions.php

This PHP script contains the shells of the implementation of the functions that were defined in the DB2Health_Report.wsdl file and registered in the db2soa.soap.server.php page. While they do not contain the full implementation of the functions, they accept the required inputs, call the necessary processing functions, and return the data in the desired format.

DB2Health_Report.wsdl

This is the WSDL document required by SOA applications in order to allow for a Web Service Consumer of a possibly varying software stack to communicate with the Web service.

Web Service Producer DB

All files grouped into this category involve the implementation of the Web service functions whose shells were exposed in the above category. This includes all classes to connect and query a DB2 database, those related to validating Web service inputs from the consumer, and generating error messages where applicable.

PHP
  • class.db.worker.connection.php
  • class.db.worker.query.php
  • class.db.render.php
  • class.db.error.php

class.db.worker.connection.php

This PHP class is used to establish and free connections to the DB2 database.

class.db.worker.query.php

This PHP class is used to perform the desired XQuery statements on the database regarding health alert information. It is the XQuery statement, processed using the DB2 database engine, that returns the data in the desired XML schema format. No PHP DOM parsing is used to do this. It is also responsible for freeing memory after the query results have been obtained.

class.db.render.php

This PHP class is a centralized mechanism through which DB2 database connections are established, the appropriate queries are executed (depending on the requested Web service function), connections are freed, and XML data is returned. Because it accepts the Web service function inputs, it also performs input validation as a security mechanism.

class.db.error.php

This PHP class serves as a self-contained error object with which all Web Service Producer errors can be fully described. It defines all of the error codes and messages that the Web Service Producer is capable of returning. To ensure server consistency, the error objects contain an XML string representation as defined in the Web service WSDL. By following this example, the consumer only has to recognize a single type of XML document format. It will also log all XML error messages to the server disk.
